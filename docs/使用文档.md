# ByteMsg233 使用文档

## 命名规范

产品和协议品牌名统一写作 **ByteMsg233**。文档、协议导出、benchmark、页面 UI 和发布文案都必须使用 `ByteMsg233` 作为展示名。小写 `bytemsg233` 只用于 CLI 命令、Go module path、包名、仓库名和文件路径。

## 安装

```bash
# macOS / Linux
curl -fsSL https://raw.githubusercontent.com/neko233-com/bytemsg233/main/scripts/install.sh | bash

# Windows PowerShell
irm https://raw.githubusercontent.com/neko233-com/bytemsg233/main/scripts/install.ps1 | iex

# Go toolchain
go install github.com/neko233-com/bytemsg233/cmd/bytemsg233@latest
```

## 60 秒开始

```bash
bytemsg233 init game
bytemsg233 compile game.bmsg.json -l go,csharp,typescript,rust,java -o ./gen
bytemsg233 export game.bmsg.json -f md,html,bmsg -o ./protocol
```

默认 DSL 是 JSON，默认文件名是 `.bmsg.json`。它的定位是替代 Protobuf 的 `.proto`：写协议、生成代码、让每个语言用自己的风格接入。

YAML 仍然支持；legacy `.bmsg` 也支持解析和导出，后续可以用于插件或更紧凑的手写 DSL 实验。

## JSON DSL

顶层 key 就是消息名，value 就是这个消息的结构。`schema`、`package`、`enums` 是保留 key。

字段默认只需要写字段名、类型、可选 comment。`tag` 不写时按 JSON 字段声明顺序自动生成；`packetId` 是消息级可选字段，用于游戏协议路由。

`protocolVersion` 是业务协议版本号。生成代码会导出 `ByteMsgProtocolVersion`，业务侧可以从生成包/库里读取版本，而不是让每条消息都携带版本号。推荐 server/client 在 socket/session 建立时发送一次 `ProtocolHello(version, minCompatible)`，通过后本连接不再给每条消息重复加版本号；如果网关要在同一连接混跑多协议，再把版本放进业务 envelope。内容指纹由业务侧自己决定是否需要，ByteMsg233 runtime 默认不生成、不强制校验。

生成 reader 和运行时会跳过支持 wire type 的未知字段，所以新增字段不会破坏旧 reader。旧服务重新编码时不会自动保留未知字段，除非业务层显式保存它们。

能做二进制生成的目标会导出协议版本工具方法，例如 `GetByteMsg233ProtocolVersion()` 或目标语言等价 API。Go 快路径还会导出 `IByteMsg233Api`，包含 `SerializeByteMsg233()` 和 `DeserializeFromByteMsg233(data)` 两个接口化入口；游戏热路径仍然可以直接使用更底层的 append/decode API，避免额外抽象影响性能。

```json
{
  "schema": "bymsg/v1",
  "protocolVersion": 7,
  "package": "com.example.game",
  "enums": {
    "HeroState": ["IDLE", "MOVING", "DEAD"]
  },
  "Hero": {
    "packetId": 1001,
    "comment": "Hero profile",
    "id": {
      "type": "uint32",
      "comment": "Hero ID"
    },
    "name": {
      "type": "string",
      "comment": "Hero name"
    },
    "skill_ids": { "list": "uint32", "comment": "Skill IDs" },
    "attrs": { "map": ["string", "uint32"], "comment": "Attributes" },
    "state": "HeroState"
  }
}
```

也支持显式 `messages` 写法：

```json
{
  "schema": "bymsg/v1",
  "package": "com.example.game",
  "messages": {
    "Hero": {
      "fields": {
        "id": { "type": "uint32", "tag": 1 },
        "name": { "type": "string", "tag": 2 }
      }
    }
  }
}
```

### 字段规则

| 字段 | 说明 |
|---|---|
| `type` | 字段类型，支持标量、`list<T>`、`map<K,V>` 和 enum |
| `tag` | wire tag，可选；默认按 JSON 字段顺序生成 |
| `comment` | 简写注释，会同时用于默认中英文注释 |
| `description.zh` | 中文注释 |
| `description.en` | 英文注释 |

### Enum

enum 可以用数组自动编号：

```json
{
  "enums": {
    "MoveState": ["IDLE", "RUN", "DASH"]
  }
}
```

也可以显式写值：

```json
{
  "enums": {
    "ErrorCode": {
      "OK": 0,
      "TIMEOUT": 10
    }
  }
}
```

带注释：

```json
{
  "enums": {
    "ItemQuality": {
      "comment": "Item quality",
      "values": ["WHITE", "GREEN", "BLUE"]
    }
  }
}
```

### List / Map

list 和 map 可以继续用字符串，也可以用 JSON 结构，后者更适合写注释：

```json
{
  "Hero": {
    "skill_ids": { "list": "uint32", "comment": "Skill IDs" },
    "attrs": { "map": ["string", "uint32"], "comment": "Attributes" },
    "inventory": { "map": { "key": "uint32", "value": "ItemQuality" } }
  }
}
```

协议建模建议：

- 优先引用消息 class，比如 `"profile": "PlayerProfile"`。
- 不要在字段里嵌套匿名消息结构；游戏通信协议更适合扁平 class + 引用。
- 默认空配置就是最优路径：不写 tag、不写额外 card/包装结构。
- 池化和 reset 是一等公民，生成代码默认面向低 GC 或无 GC 的客户端循环。
- 目标语言 lib 的 encode/decode 与池化路径必须完全单线程：不使用锁、并发集合、channel、goroutine、线程池、后台 worker 或原子同步。

### 类型

| 类型 | 用途 |
|---|---|
| `bool` | 布尔值 |
| `int32` / `int64` | 有符号整数 |
| `uint32` / `uint64` | 无符号整数 |
| `sint32` / `sint64` | ZigZag 编码有符号整数 |
| `float32` / `float64` | 浮点数 |
| `string` | UTF-8 文本 |
| `bytes` | 二进制 |
| `list<T>` | 列表 |
| `map<K,V>` | 字典 |
| enum name | 枚举字段 |

## CLI

### init

```bash
bytemsg233 init <name>
```

生成 `<name>.bmsg.json`。

### compile

```bash
bytemsg233 compile <schema.bmsg.json> [flags]
```

| 参数 | 简写 | 默认值 | 说明 |
|---|---|---:|---|
| `--lang` | `-l` | `go` | 目标语言 |
| `--output` | `-o` | `.` | 输出目录 |
| `--locale` | | `en` | 注释语言，支持 `en` / `zh` |

```bash
bytemsg233 compile protocol.bmsg.json -l go -o ./gen/go
bytemsg233 compile protocol.bmsg.json -l csharp --locale zh -o ./Assets/Generated
bytemsg233 compile protocol.bmsg.json -l go,csharp,typescript,rust,java -o ./gen
```

### export

```bash
bytemsg233 export protocol.bmsg.json -f md,html,bmsg -o ./protocol
```

| 格式 | 输出 | 用途 |
|---|---|---|
| `md` | `protocol.md` | 给人看的协议接入说明 |
| `html` | `protocol.html` | 可直接发给用户/客户端同学看的独立页面 |
| `bmsg` | `protocol.bmsg` | legacy/未来扩展 DSL |

### install-lib

```bash
bytemsg233 install-lib <language> --to <target-dir>
```

这个命令从主仓库的 `libs/<language>` 复制运行时库到用户项目，不依赖包管理发布。

```bash
bytemsg233 install-lib csharp --to ../UnityGame/Assets/Plugins/ByteMsg233
bytemsg233 install-lib typescript --to ../web/vendor/bytemsg233
bytemsg233 install-lib go --to ../server/third_party/bytemsg233
bytemsg233 install-lib rust --to ../rust-game/vendor/bytemsg233
bytemsg233 install-lib java --to ../android/libs/bytemsg233
bytemsg233 install-lib cpp --to ../native/third_party/bytemsg233
bytemsg233 install-lib c --to ../embedded/vendor/bytemsg233
```

如果不在 bytemsg233 仓库根目录执行，可以传 `--repo`：

```bash
bytemsg233 install-lib csharp --repo D:/Code/bytemsg233 --to ./Assets/Plugins/ByteMsg233
```

## 语言优先级

| 语言 | 目标 |
|---|---|
| Go | 服务端、工具链、轻量运行时 |
| C# / Unity | Unity client，同一个 namespace 接入 |
| TypeScript / JavaScript | Web、Vite、Next.js、React/Vue 前端工程 |
| Rust | 高性能客户端、服务端和游戏工具 |
| Java | JVM、Android、后端项目 |
| C++ | 原生客户端、游戏引擎、性能敏感工具 |
| C | embedded、FFI、最小运行时宿主 |
| Kotlin | Android / KMP |
| Swift | iOS / macOS |
| Dart / Flutter | Flutter 客户端 |
| Lua | 游戏脚本、热更层、运行时 glue |

Python 仍然支持生成，偏工具和测试场景。官方语言仓库和 API 规范见 [`docs/LANGUAGES.md`](LANGUAGES.md)。

## 生成代码体验

| 语言 | 对象池 | Enum |
|---|---|---|
| Go | `AcquireHero()` / `ReleaseHero()` | typed consts + parse helper |
| C# / Unity | `new Hero()` / `Hero.Prewarm()` / `Hero.Rent()` / `Hero.Return()` / `Release()` | `partial class` + native enum helper |
| TypeScript | `Hero.acquire()` / `release()` | `enum` + helper namespace |
| Rust | `ByteMsgPool<T>` | `enum` + `from_value()` |
| Java | `Hero.acquire()` / `release()` | enum class + `fromValue()` |
| C++ | `HeroPool.Acquire()` / `Release()` | `enum class` |
| C | caller-owned pool/context | generated constants |
| Kotlin / Swift / Dart | language-native constructor + pool helper | native enum |

Go 生成消息支持调试文本输出，适合工具、日志、fixture 检查：

```go
text := hero.ByteMsgText()
dst := hero.AppendByteMsgText(buf[:0])
```

调试文本使用 schema 字段名，但它不是协议格式，也不参与前后端通信。热路径必须保持完全二进制，优先 caller-owned buffer 和池化对象。

C# / Unity 生成消息默认是 `partial class`，用户扩展写到单独 partial 文件即可，重新生成协议代码不覆盖业务扩展。运行时可以普通 `new Hero()`，也可以在加载阶段 `Hero.Prewarm(count)`，之后用 `Hero.Rent()` / `Hero.Return(hero)` / `hero.Release()` 走池化热路径。

单文件语言默认导出文件名为 `ByteMsg233_Export` 加语言扩展名，例如 `ByteMsg233_Export.go`、`ByteMsg233_Export.cs`、`ByteMsg233_Export.ts`。

## YAML 兼容

JSON 是主 DSL，但 YAML 继续可用：

```yaml
schema: bymsg/v1
package: com.example.game

messages:
  Hero:
    fields:
      id:
        type: uint32
        tag: 1
```

YAML 适合已有流水线；新文档和 `init` 默认使用 JSON。

## 性能

这张表不是为了证明某个小对象“刚好赢”。它覆盖小包、重复 DTO、战斗输入和 100 人排行榜。真正的游戏协议经常是一堆重复结构：背包、邮件、任务、排行榜、战斗帧。字段名如果每条都发一次，数据量很快就上去了。

| 场景 | ByteMsg233 | Protobuf | JSON payload | MessagePack |
|---|---:|---:|---:|---:|
| 玩家信息，10 fields | 61 B | 61 B | 173 B | 155 B |
| 聊天消息，5 fields | 57 B | 57 B | 116 B | 103 B |
| ChatDto 全类型 | 304 B | 316 B | 647 B | 531 B |
| RPC envelope + ChatDto，1x | 316 B | 328 B | 928 B | 597 B |
| 战斗输入，10 人 | 130 B | 266 B | 1097 B | 931 B |
| TaskDto 列表，100 条 | 2261 B | 4044 B | 14691 B | 13303 B |
| 排行榜，100 人 | 2518 B | 3608 B | 9602 B | 8711 B |

还需要验证 ChatDto 全类型、登录全量下发、实时战斗帧、排行榜、大状态快照等包型。ChatDto 专门覆盖 bool、有符号/无符号整数、float、double、string、bytes、list、map 参数 KV 和嵌套自定义类型。只测一个很小的 DTO 没意义，也不公平。

```bash
go test ./pkg/binary/... -run "TestBenchmark_SizeComparison" -v
go test ./pkg/binary/... -run "TestBenchmark_ChatDtoAllTypesRoundTrip" -v
go test ./pkg/binary/... -run "TestGame_" -v
go test ./pkg/binary/... -bench="Benchmark(Encode|Decode)_" -benchmem
go test ./pkg/binary/... -bench="Benchmark(Encode|Decode)_ChatDtoAllTypes" -benchmem
go test ./pkg/binary/... -bench="BenchmarkGame_" -benchmem
```

游戏消息结构规则见 [`docs/GAME_BINARY.md`](GAME_BINARY.md)，完整 benchmark 说明见 [`docs/BENCHMARK.md`](BENCHMARK.md)。
