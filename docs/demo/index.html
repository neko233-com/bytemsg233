Install first. Ship schema next.

ByteMsg233 turns one JSON schema into native-feeling Go, C#, Java, TypeScript, Rust, and Python models with single-threaded pools, enum helpers, localized comments, compact binary payloads, and 0-GC hot paths.

The DSL is JSON because big protocols need structure.

When messages grow, JSON keeps message names, fields, enum maps, and generated diffs predictable. YAML remains supported; JSON is the native contract.

One file. Structured comments. Stable diffs.

Descriptions live as fields, not line comments. That lets class comments and field comments generate cleanly in every target language and locale.

message keyTop-level keys such as `Hero` and `InventoryItem` map to native classes or structs.
fieldsEvery field declares `type`, `tag`, and optional localized descriptions.
enumsEnum values generate native enum helpers instead of forcing raw integers.
exportsGenerate `.md` for people and `.bmsg` for legacy/future DSL tooling.
CLIschema to SDK
# create user.bmsg.json
bytemsg233 init user

# generate all supported languages
bytemsg233 compile user.bmsg.json \
  -l go,csharp,typescript,rust,java \
  -o ./gen

# export docs and extension-friendly .bmsg
bytemsg233 export user.bmsg.json \
  -f md,bmsg \
  -o ./protocol

Generated code should feel native, not translated.

Each target gets its own pool pattern, enum helpers, comments, and naming style. The schema is shared; the developer experience is language-specific.

Go

Struct tags, exported fields, and single-threaded slice-stack pool helpers.

hero := AcquireHero() defer ReleaseHero(hero) hero.State = HeroStateMOVING

C# / Unity

Partial classes, XML docs, native collections, and Unity-friendly pooling.

var hero = Hero.Rent(); hero.State = HeroState.MOVING; hero.Release();

Java

Javadoc, getters/setters, enum value lookup, queue-backed pooling.

Hero hero = Hero.acquire(); hero.setState(HeroState.MOVING); hero.release();

TypeScript

Runtime JS plus `.d.ts`, suitable for web and frontend build systems.

const hero = Hero.acquire(); hero.state = HeroState.MOVING; hero.release();

Python

Dataclasses, IntEnum, type hints, and a small class-level object pool.

hero = Hero.acquire() hero.state = HeroState.MOVING hero.release()

Rust

Traits, explicit pools, and zero-copy friendly buffer APIs.

let mut hero = pool.acquire(); hero.state = HeroState::Moving; pool.release(hero);

C++

Header/runtime split for engines, tools, and native clients.

auto hero = pool.Acquire(); hero->state = HeroState::Moving; pool.Release(hero);

C

Caller-owned buffers and contexts for embedded or FFI hosts.

ByteMsgHero hero; bytemsg_hero_init(&hero); bytemsg_hero_reset(&hero);

Kotlin / Swift

Mobile-native constructors, extension methods, and pool helpers.

val hero = Hero.acquire() hero.state = HeroState.MOVING hero.release()

Go

Use sidecar methods in the same package for custom behavior. Pool flow should reuse message storage.

hero := bytemsg233.AcquireHero()
defer bytemsg233.ReleaseHero(hero)

hero.Id = 1001
hero.State = bytemsg233.HeroStateMOVING

buf := make([]byte, 0, 256)
buf, err := hero.MarshalByteMsgAppend(buf)

C# / Unity

Generated messages are partial classes. Prewarm during loading, then rent and release in gameplay hot paths.

Hero.Prewarm(256);

var hero = Hero.Rent();
hero.Id = 1001;
hero.State = HeroState.MOVING;

writer.Reset(buffer);
hero.WriteTo(ref writer);
hero.Release();

TypeScript

Use normal constructors for tools and acquire/release for hot client loops.

const hero = Hero.acquire();
hero.id = 1001;
hero.state = HeroState.MOVING;

const writer = ByteMsgWriter.from(buffer);
hero.writeTo(writer);
hero.release();

Rust

Keep extension traits outside generated files and reuse Vec-backed buffers for encode/decode.

let mut hero = pool.acquire();
hero.id = 1001;
hero.state = HeroState::Moving;

buffer.clear();
hero.write_to(&mut buffer)?;
pool.release(hero);

C++

Use generated headers plus a small runtime. Engines can bind pools to arenas or frame allocators.

auto hero = heroPool.Acquire();
hero->id = 1001;
hero->state = HeroState::Moving;

ByteMsgWriter writer(buffer);
hero->WriteTo(writer);
heroPool.Release(hero);

C

Use caller-owned structs, buffers, and decode context. No hidden allocations in hot paths.

ByteMsgHero hero;
bytemsg_hero_init(&hero);
hero.id = 1001;
hero.state = BYTEMSG_HERO_STATE_MOVING;

ByteMsgWriter writer;
bytemsg_writer_init(&writer, buffer, capacity);
bytemsg_hero_write(&hero, &writer);

Kotlin

Kotlin gets native constructors, companion pool helpers, and extension functions for user code.

Hero.prewarm(256)

val hero = Hero.acquire()
hero.id = 1001u
hero.state = HeroState.MOVING

writer.reset(buffer)
hero.writeTo(writer)
hero.release()

Swift

Swift uses extension methods for custom behavior and explicit pool objects for reusable messages.

let hero = heroPool.rent()
hero.id = 1001
hero.state = .moving

writer.reset(buffer)
try hero.write(to: writer)
heroPool.return(hero)
